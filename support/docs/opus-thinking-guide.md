# 🧠 Opus 4.5 Thinking 모드 설정 가이드

> "Opus 4.5는 지금까지 써본 코딩 모델 중 최고. Sonnet보다 느리지만 개입이 적어도 되고 툴 사용이 뛰어나서 결국 전체 속도는 더 빠르다" - Boris Cherny

## 왜 Opus 4.5인가?

| 모델 | 속도 | 정확도 | 개입 필요 | 툴 사용 | 총평 |
|------|------|--------|----------|---------|------|
| Sonnet 4.5 | ⚡⚡⚡ | ⭐⭐⭐ | 자주 필요 | 보통 | 간단한 작업 |
| **Opus 4.5** | ⚡⚡ | ⭐⭐⭐⭐⭐ | 거의 없음 | 뛰어남 | **복잡한 작업 권장** |
| Haiku 4.5 | ⚡⚡⚡⚡⚡ | ⭐⭐ | 매우 자주 | 기본 | 초간단 작업만 |

## Project Sunshine에서 Opus 4.5 사용

### 1. 모델 설정
```bash
# Claude Code에서 모델 변경
claude --model opus-4.5

# 또는 settings.json에서 설정
{
  "model": "opus-4.5",
  "thinking": true
}
```

### 2. Thinking 모드 활성화

Thinking 모드를 켜면 Claude가 작업 전에 깊이 생각합니다:

```bash
# CLI에서 thinking 모드 활성화
claude --thinking

# 또는 대화 중 활성화
> /model opus-4.5 --thinking
```

### 3. 언제 Opus 4.5를 사용하는가?

#### ✅ Opus 4.5 사용 권장
- 전체 파이프라인 실행
- 새로운 에이전트 개발
- 복잡한 버그 수정
- 아키텍처 설계
- 여러 파일 동시 수정
- MCP 연동 작업

#### ⚡ Sonnet으로 충분한 경우
- 단순 오타 수정
- 문서 업데이트
- 간단한 설정 변경
- 단일 파일 수정

## Thinking 모드 활용법

### 기본 사용
```
Claude, cherry 주제로 전체 파이프라인 실행해줘. 깊이 생각해서 진행해.
```

### 복잡한 문제 해결
```
이 버그를 깊이 분석해줘:
1. 원인이 뭘까?
2. 어떤 파일들이 영향 받나?
3. 최선의 해결책은?
4. 부작용은 없나?
```

### 아키텍처 결정
```
새로운 스케줄러 에이전트를 설계해야 해.
기존 시스템과 어떻게 연동할지 깊이 생각해줘.
```

## Project Sunshine 최적 설정

### settings.json
```json
{
  "model": {
    "default": "opus-4.5",
    "thinking": true,
    "fallback": "sonnet-4.5"
  },
  "autoSwitch": {
    "simpleTask": "sonnet-4.5",
    "complexTask": "opus-4.5"
  }
}
```

### 작업별 모델 선택 가이드

| 작업 | 권장 모델 | 이유 |
|------|----------|------|
| 전체 파이프라인 | Opus 4.5 + Thinking | 9개 에이전트 조율 필요 |
| 이미지 프롬프트 생성 | Opus 4.5 | 창의성 + 정확성 |
| 단순 텍스트 수정 | Sonnet 4.5 | 속도 우선 |
| 버그 수정 | Opus 4.5 + Thinking | 깊은 분석 필요 |
| 코드 리뷰 | Opus 4.5 | 맥락 이해 중요 |
| 문서 작성 | Sonnet 4.5 | 속도 우선 |

## 비용 vs 효율 분석

### 시나리오: cherry 주제 콘텐츠 생성

**Sonnet 4.5로 진행 시:**
- 실행 시간: 5분
- 개입 횟수: 3-4회
- 수정 필요: 2-3회
- 총 소요 시간: ~15분

**Opus 4.5 + Thinking으로 진행 시:**
- 실행 시간: 8분
- 개입 횟수: 0-1회
- 수정 필요: 0-1회
- 총 소요 시간: ~10분

> 💡 **결론:** Opus가 느려 보이지만 개입/수정 시간을 줄여서 결국 더 빠름

## 팁

### 1. Thinking 프롬프트 패턴
```
"깊이 생각해서..."
"모든 가능성을 고려해서..."
"단계별로 분석해서..."
"잠재적 문제를 미리 파악해서..."
```

### 2. 복잡한 작업 전 명시
```
이 작업은 복잡하니까 Opus 4.5로 진행하고,
실행 전에 계획을 먼저 세워줘.
```

### 3. 자동 모델 전환 활용
```bash
# 간단한 작업은 자동으로 Sonnet 사용
claude "오타 수정해줘"  # → Sonnet

# 복잡한 작업은 자동으로 Opus 사용
claude "전체 리팩토링해줘"  # → Opus
```

## 주의사항

1. **API 비용**: Opus는 Sonnet보다 비쌈 (약 5배)
2. **속도**: 단일 요청은 Sonnet이 빠름
3. **선택 기준**: 작업 복잡도로 판단, 속도로 판단하지 않기
